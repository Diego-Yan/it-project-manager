## 二、问题详情与修复

### #1 [P1] 队列 worker 未调度 — webhook 通知永不投递

**文件：** `routes/console.php`
**问题：** `.env` 和 `.env.example` 均配置 `QUEUE_CONNECTION=database`。`NotificationService::send()` 通过 `SendWebhookNotification::dispatch()` 将 Job 投递到 database 队列表。但 `routes/console.php` 中**没有任何 `queue:work` 调度**，项目中也无 systemd/supervisor 常驻 worker 配置。

**影响：** 所有 webhook 通知（工单创建/分配/解决/关闭、项目完成/逾期、任务分配/完成、成员加入等）**永远不会被消费执行**。`jobs` 表无限堆积。整个通知系统形同虚设——用户在系统中操作后，企微/钉钉群收不到任何通知。

**修复：** 添加 `queue:work --stop-when-empty --max-time=60` 每分钟调度。`--stop-when-empty` 确保处理完积压任务后自动退出，避免长驻进程的内存泄漏风险（SQLite + 小型部署的推荐方案）。配合 `withoutOverlapping()` 防止多个 worker 同时运行。

```php
Schedule::command('queue:work --stop-when-empty --max-time=60')
    ->everyMinute()
    ->withoutOverlapping()
    ->appendOutputTo(storage_path('logs/queue-worker.log'));
```

### #2 [P2] TaskManager saveTask() 编辑权限绕过

**文件：** `app/Livewire/Projects/TaskManager.php`
**问题：** `editTask()` 方法在加载编辑表单前检查了 `canManageTask()` 权限。但 `saveTask()` 在编辑分支（`$this->editingTask && $this->editingTaskId`）中**直接执行 `$task->update($data)` 而不重新校验权限**。

Livewire 的 public 属性（`editingTaskId`、`editingTask`）可被客户端篡改。攻击路径：
1. 调用 `editTask()` 操作一个自己有权限管理的任务（通过权限检查）
2. 篡改 `editingTaskId` 指向一个无权管理的任务
3. 调用 `saveTask()` → 绕过权限修改该任务

**修复：** 在 `saveTask()` 编辑分支重新加载任务（带 `project_id` 作用域防跨项目篡改）并调用 `canManageTask()` 校验。

### #3 [P2] 生产环境异常处理吞掉 ValidationException

**文件：** `bootstrap/app.php`
**问题：** 原异常处理 `render` 回调在生产环境对"非 AuthenticationException"的所有异常**一律返回 500 页面**：

```php
if (app()->isProduction() && !($e instanceof AuthenticationException)) {
    // 404 → 404 页面
    // HttpException → 对应错误页
    // 其他 → 500 页面 ← ValidationException 落入此分支
    return response()->view('errors.500', [], 500);
}
```

`ValidationException` 不是 `HttpException`，也不是 `AuthenticationException`，因此在生产环境中**表单验证错误会显示 500 服务器错误页面**，而非字段级验证反馈。

虽然 Livewire 组件内部的验证会被 Livewire 拦截处理（不经过全局异常处理器），但非 Livewire 路径（如 API 端点 BotController 的验证、未来可能添加的 API 路由）仍受影响。

**修复：** 重构异常处理逻辑，按异常类型分层处理：
- 非生产环境：返回 `null`（交给 Laravel 默认处理器，显示完整调试信息）
- AuthenticationException：返回 `null`（Laravel 处理重定向到登录页）
- ValidationException：返回 `null`（Livewire/Laravel 处理字段级反馈）
- HttpException 5xx：渲染 500 页面
- HttpException 4xx：尝试渲染对应错误页，无模板则交给默认处理器
- NotFoundHttpException：渲染 404 页面
- 其他异常：渲染 500 页面（防止堆栈泄露）

### #4 [P3] SendWebhookNotification 重试机制失效

**文件：** `app/Jobs/SendWebhookNotification.php`
**问题：** Job 配置了 `$tries = 3`（重试 3 次）和 `failed()` 方法（投递失败后通知管理员）。但 `handle()` 方法中**每个 webhook 的 HTTP 调用都被 `try/catch` 吞掉**，只记 warning 日志。

**影响：**
- Job 永远"成功"完成 → `failed()` 方法**永远不会被触发** → 管理员收不到投递失败通知
- `$tries = 3` 重试机制**形同虚设** → 即使所有 webhook 都失败也不会重试
- 配合 #1（无 worker 调度），形成了双重失效：即使有 worker，失败的 webhook 也不会重试

**修复：** 用 `$failures` 计数器收集失败结果。若全部 webhook 投递失败则抛出 `RuntimeException`，触发 `$tries` 重试和 `failed()` 管理员通知。部分成功时不抛出（已投递成功的 webhook 不应重复投递）。

### #5 [P2] CheckDeadlines 通知轰炸 — 无去重机制

**文件：** `app/Console/Commands/CheckDeadlines.php`
**问题：** `check:deadlines` 调度为每天 9:00 和 15:00 各执行一次。每次执行时，**所有逾期项目和即将到期项目都会收到通知**，无任何去重机制。

**影响：**
- 逾期项目：每天收到 **2 次**通知（9:00 + 15:00），每周 **14 次**
- 即将到期项目：同上，3 天窗口内收到 **6 次**重复通知
- 站内通知铃铛被淹没，用户开始忽略通知 → 通知失去意义

**修复：** 用 `Cache` 按天去重。每个项目/任务的 `dedupKey` 格式为 `deadline_notified:{type}:{id}:{YYYY-MM-DD}`，TTL 到当天结束。当天已通知的项目在后续执行中跳过，确保每天最多 1 次通知。

### #6 [P3] AssetManager 权限拒绝静默返回

**文件：** `app/Livewire/Itsm/AssetManager.php`
**问题：** `delete()` 和 `deleteCatalogItem()` 方法在权限不足时静默 `return`，不设置任何 `session()->flash()` 消息。用户点击删除按钮后页面无任何反馈，无法区分"删除成功"和"权限不足被拒绝"。

同一文件中 `save()`、`edit()`、`addCatalogItem()` 方法均使用了 `session()->flash('error', ...)` 给出明确反馈，但 `delete()` 系列方法未遵循此模式。

**修复：** 统一为 `session()->flash('error', __('没有资产管理权限'))` + `return`，与同类方法保持一致。`delete()` 额外补充成功提示。

## 五、审查中确认良好的部分（无需修改）

- **队列配置**：`after_commit: true` 确保 DB 事务提交后才投递 Job，避免事务回滚后 Job 仍执行
- **SQLite 并发策略**：AssetManager 消耗品 asset_tag 生成根据驱动选择 `lockForUpdate`(MySQL) 或事务(SQLite)
- **Dashboard 查询优化**：4 次 COUNT → 1 次 GROUP BY，3 次 COUNT → 1 次 selectRaw
- **SidebarComposer 缓存**：6 次独立 COUNT 合并为 1 次缓存查询，TTL=5 分钟，状态变更时主动 flush
- **TaskManager 原子认领**：`claimTask()` 用 WHERE 条件原子更新防竞态
- **IncidentTimeline 状态守卫**：只允许 resolved → closed，mitigated 需先 resolve
- **Notification 模型**：静态 `send()` 方法简洁，type 字段支持颜色映射
- **CSP 安全头**：SecurityHeaders 中间件配置完善，HSTS 仅 HTTPS 生效

## 七、后续建议（第4轮可关注）

1. **SLA Seeder 缺失**：DatabaseSeeder 未创建 SLA 初始数据，`Sla::getDeadline()` 在未配置时返回 null，SLA 违约检测完全失效
2. **AiChat LLM 调用限流**：`send()` 无频率限制，用户可频繁调用外部 API 导致成本失控
3. **消耗品 asset_tag 生成**：`max('asset_tag')` 在超过 9999 条时因字典序比较会导致编号重复（CON-9999 > CON-10000 字典序）
4. **知识库附件公开访问**：附件存储在 public disk，知道 URL 即可无认证访问
5. **PHP Enum 类引入**：建议用 PHP 8.1 backed enum 替代散落在多处的字符串枚举（priority/type/status），从根本上防止不一致问题再次发生
6. **队列清理调度**：建议添加 `queue:prune-failed` 定时清理 failed_jobs 表，防止无限增长

## 二、问题详情与修复

### #1 [P1] UserManager 创建用户权限提升漏洞

**文件：** `app/Livewire/Admin/UserManager.php`
**问题：** `saveUser()` 方法在编辑分支检查了 `edit users` 权限，但 **AD 创建分支和本地用户创建分支均未检查 `create users` 权限**。用户管理路由仅需 `view users`（只读）权限即可加载页面，而 Livewire 的 action 方法通过 AJAX 调用，路由级中间件仅保护页面加载。因此任何有 `view users` 权限的用户（如 IT 主管角色拥有 `view users` 但无 `create users`）都能越权创建任意用户，包括本地管理员账号。

**修复：** 在 `saveUser()` 开头统一校验——编辑检查 `edit users`，创建检查 `create users`。

```php
// [REVIEW-FIX-R1 #1 P1] 权限提升修复
if ($this->isEditing) {
    if (!auth()->user()->can('edit users')) abort(403);
} else {
    if (!auth()->user()->can('create users')) abort(403);
}
```

### #2 [P2] AiChat 审查注释泄漏到运行时字符串

**文件：** `app/Livewire/AiChat.php` 第 137 行
**问题：** `buildContext()` 中拼接任务上下文时，误将审查注释 `// [REVIEW-FIX] SP12.5: null-safe project access` 作为字符串的一部分拼接到了发送给 LLM 的实际内容中。该注释会随上下文发送给外部 LLM，并最终展示给用户。

**修复：** 移除注释拼接，仅保留业务字符串。

---

### #3 [P2] EnvService .env 写入竞态条件 (TOCTOU)

**文件：** `app/Services/EnvService.php`
**问题：** 原实现先用 `file()` 无锁读取 .env 内容，解析构建新行后，再用 `fopen('w')` + `flock(LOCK_EX)` 写入。**读取发生在加锁之前**，两个并发请求可能同时读到旧内容、各自构建新行、先后写入，导致后写者覆盖先写者的更新（丢失更新）。AD/IM/AI 三个配置页面并发保存时存在此风险。

**修复：** 改为 `fopen('r+')` → `flock(LOCK_EX)` → **锁内** `stream_get_contents` 读取最新内容 → 解析构建 → `ftruncate(0)` + `rewind` + `fwrite` 重写。读取和写入均在排他锁保护下完成。

---

### #4 [P2] db:backup 调度未清理旧备份

**文件：** `routes/console.php`
**问题：** `DbBackup` 命令支持 `--prune` 选项清理 30 天前旧备份，但调度器调用时未传该选项，导致 `storage/app/backups/` 目录每日新增一个备份文件且永不清理，长期运行将写满磁盘。

**修复：** 调度改为 `Schedule::command('db:backup --prune')`。

---

### #5 [P2] 本地登录缺少审计日志

**文件：** `app/Http/Controllers/Auth/LoginController.php`
**问题：** AD 登录有完整的成功/失败日志，但**本地登录成功和失败均无任何日志记录**。安全审计链断裂，无法追溯本地账号的暴力破解探测、撞库尝试或异常登录。

**修复：** 补充本地登录失败（含 IP）、账号禁用拒绝、登录成功三个审计日志点，与 AD 登录保持一致。日志仅记录用户名和 IP，不记录密码。

---

### #6 [P3] ProjectForm 缺少组件内权限校验

**文件：** `app/Livewire/Projects/ProjectForm.php`
**问题：** 路由层有 `can:create/edit projects` 中间件保护页面加载，但 `save()` 方法内部未做权限校验。虽非可利用漏洞（路由中间件在 Livewire 后续请求会重新执行），但不符合纵深防御原则。

**修复：** 在 `save()` 开头补充内部权限校验，编辑检查 `edit projects`，创建检查 `create projects`。

---

### #7 [P1] ProjectForm $messages 属性 fatal error

**文件：** `app/Livewire/Projects/ProjectForm.php` 第 81 行
**问题：** **预先存在的工作区 bug**（git 确认 HEAD 原始版本为纯字符串字面量，工作区被改为 `__()` 调用）。`$messages` 属性的值使用了 `__('...')` 函数调用，但 PHP **不允许函数调用作为属性默认值（常量表达式）**，导致 `Constant expression contains invalid operations` fatal error。**创建/编辑项目页面会直接崩溃无法访问。**

**修复：** 将 `protected $messages = [...]` 属性改为 `protected function messages(): array` 方法（Livewire 原生支持），方法体内可合法调用 `__()`。

---

### #8 [P1] 同类 $属性 __() fatal error（3 个文件）

**文件：** `app/Livewire/Projects/LinkManager.php`、`app/Livewire/Admin/RoleManager.php`、`app/Livewire/Admin/WebhookManager.php`
**问题：** 与 #7 同类问题，均为属性默认值使用 `__()` 函数调用导致 fatal error：
- **LinkManager**：`$messages` 属性
- **RoleManager**：`$systemRoles`、`$permGroups`、`$permLabels` 三个 protected 属性
- **WebhookManager**：`$availableEvents` public 属性

这三个组件（项目关联管理、角色管理、Webhook 管理）加载即崩溃。

**修复策略：**
- `$messages` 属性 → 改为 `messages()` 方法（LinkManager）
- protected/public 属性 → 声明为空数组，在 `boot()` 方法中初始化（RoleManager、WebhookManager）。`boot()` 在每次 Livewire 请求均执行，确保属性可用。

## 五、审查中确认良好的部分（无需修改）

- **安全响应头**：SecurityHeaders 中间件配置完善（CSP/X-Frame-Options/HSTS 等）
- **LDAP 注入防护**：`ldap_escape()` 正确使用，过滤器转义到位
- **Zabbix/Webhook Token 加密**：`ZabbixConfig` 使用 `Crypt::encryptString`，含旧数据兼容
- **Bot 回调签名验证**：企微/钉钉 webhook 均有 `hash_equals` 时序安全比较
- **ITSM 组件权限覆盖**：TicketBoard/ChangeManager/IncidentManager 等写操作均有内部 `can()` 校验
- **N+1 查询防护**：关键列表查询使用 `with()` 预加载
- **事务一致性**：ProjectForm 创建/编辑使用 `DB::transaction` 包裹
- **会话安全**：SESSION_ENCRYPT=true，登出正确 invalidate + regenerateToken
- **登录限流**：`throttle:5,1` 防暴力破解 + AD 失败锁定机制

## 六、后续建议（第2轮可关注）

5. **全面排查属性默认值**：建议增加 CI 中的 `php -l` 检查，防止同类 `__()` 属性问题再次引入

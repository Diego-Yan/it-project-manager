## 二、问题详情与修复

### #1 [P2] Ticket priority 枚举不一致 — 紧急工单无法创建

**文件：** `app/Livewire/Itsm/TicketBoard.php`、`app/Livewire/MyTickets.php`
**问题：** Ticket 优先级在系统各层存在严重不一致：

| 层 | 紧急优先级值 |
|----|-------------|
| Blade 模板下拉框（my-tickets/tickets/slas） | `critical` |
| Ticket 模型 `getPriorityLabelAttribute`/`getPriorityColorAttribute` | `critical` |
| SlaManager `orderByRaw` | `critical` |
| **TicketBoard 验证规则** | ❌ `urgent` |
| **MyTickets 验证规则** | ❌ `urgent` |

**影响：** 用户在工单表单选择"紧急"→ Blade 提交 `critical` → 验证规则只允许 `urgent` → **验证失败，紧急工单无法创建**。即使通过 BotController 创建（用 `high`），紧急优先级的 SLA 也无法正确匹配（SLA 表存 `critical`）。

**修复：** 将 TicketBoard 和 MyTickets 验证规则中的 `urgent` 统一为 `critical`，与 Blade 模板、模型 label、SLA 排序保持一致。

### #2 [P2] SlaManager formPriority 缺验证规则

**文件：** `app/Livewire/Itsm/SlaManager.php`
**问题：** `$rules` 数组只定义了 `formName`/`formResponse`/`formResolution` 的验证规则，**`formPriority` 完全没有验证**。用户可通过篡改前端请求提交任意字符串作为 SLA 的 priority 值，导致 `Sla::getDeadline($priority)` 查询永远不匹配（工单 SLA 截止时间为 null）。

**修复：** 补充 `'formPriority' => 'required|in:low,medium,high,critical'` 验证规则。

---

### #3 [P2] ProjectDetail render() pluck 字段名错误

**文件：** `app/Livewire/Projects/ProjectDetail.php` 第 226 行
**问题：** `render()` 方法中：
```php
$memberIds = $this->project->members->pluck('user_id')->toArray();
```
`$project->members` 是 `BelongsToMany` 关系，返回 **User 模型集合**，主键为 `id`。`user_id` 是 pivot 表 `project_user` 的外键，**不在 User 模型属性中**。因此 `pluck('user_id')` 返回全 `null` 数组，`whereNotIn('id', [null, null, ...])` 不生效，**已加入项目的成员仍出现在"添加成员"可选列表中**，用户尝试重复添加时会触发唯一约束错误或产生脏数据。

同一文件的 `flushMemberCaches()` 方法正确使用了 `pluck('id')`，印证了 `render()` 中的 `pluck('user_id')` 是笔误。

**修复：** `pluck('user_id')` → `pluck('id')`。

### #4 [P3] MyTickets formSource 缺 walk_in 选项

**文件：** `app/Livewire/MyTickets.php`
**问题：** `formSource` 验证规则为 `in:portal,email,phone,im_wechat,im_dingtalk`，缺少 `walk_in`（现场处理）。TicketBoard 的验证规则包含 `walk_in`。如果前端提供了 `walk_in` 选项，MyTickets 页面会验证失败。

**修复：** 补充 `walk_in` 到验证规则，与 TicketBoard 对齐。

---

### #5 [P3] Project type 常量与实际使用不一致

**文件：** `app/Models/Project.php`
**问题：** 模型定义了以下常量：
```php
public const TYPE_NEW       = 'new';
public const TYPE_MAINTENANCE = 'maintenance';  // ❌ 实际不使用
public const TYPE_UPGRADE   = 'upgrade';         // ❌ 实际不使用
```
但 ProjectForm 验证规则、`getTypeLabelAttribute`、`getTypeColorAttribute` 实际使用的值是 `new`/`improved`/`issue`。`TYPE_MAINTENANCE` 和 `TYPE_UPGRADE` 是误导性死代码，可能导致开发者引用不存在的常量值。

**修复：** 修正常量为 `TYPE_NEW='new'`、`TYPE_IMPROVED='improved'`、`TYPE_ISSUE='issue'`，与实际使用一致。

---

### #6 [P3] BotController 紧急内容优先级映射错误

**文件：** `app/Http/Controllers/Api/BotController.php`
**问题：** IM 机器人创建工单时，紧急内容（匹配 `紧急|urgent|急`）映射为 `high` 而非 `critical`。`high` 和 `critical` 是不同优先级，紧急内容应映射到最高优先级 `critical`，否则 `high` 和 `critical` 的区分失效，且 `critical` 级别的 SLA 永远无法通过 IM 渠道触发。

**修复：** `$priority = 'high'` → `$priority = 'critical'`。

## 五、审查中确认良好的部分（无需修改）

- **LDAP 注入防护**：`LdapAuthService` 全部使用 `ldap_escape()` 转义用户输入
- **SQL 注入防护**：所有 `selectRaw`/`orderByRaw` 均使用硬编码常量字符串，无用户输入拼接
- **Blade 模板 XSS 防护**：`{!! !!}` 仅用于 `__()` 翻译函数（开发者控制的静态字符串），用户数据均通过 `{{ }}` 自动转义
- **文件上传安全**：`getClientOriginalName()` 经正则过滤后仅用于显示，文件路径由 `store()` 随机生成；MIME 类型通过 `getMimeType()` 获取（非客户端提供的 Content-Type）
- **状态机一致性**：Ticket 状态转换（open→in_progress→resolved→closed）有 `canTransitionTo` 守卫，各组件统一使用 `transitionTo*` 方法
- **事务一致性**：ProjectForm/ProjectDetail 的 update+logAction 均包裹 `DB::transaction`
- **N+1 查询防护**：AiChat、TicketBoard、KnowledgeBase 等关键查询使用 `with()` 预加载
- **认证安全**：LDAP 认证通过 `ldap_bind` 验证密码（非字符串比较），有失败锁定机制
- **LIKE 通配符转义**：KnowledgeBase 搜索使用 `addcslashes($search, '%_')` 防止通配符注入

## 七、后续建议（第3轮可关注）

1. **AiChat LLM 调用限流**：`send()` 无频率限制，用户可频繁调用外部 API 导致成本失控
2. **知识库附件公开访问**：附件存储在 public disk，知道 URL 即可无认证访问（知识库本身公开，风险较低）
3. **AI 配置权限**：路由用 `can:manage roles` 权限，应有独立的 AI 配置管理权限
4. **SLA Seeder**：DatabaseSeeder 未创建 SLA 初始数据，`Sla::getDeadline()` 在未配置时返回 null，SLA 违约检测失效
5. **枚举统一治理**：建议引入 PHP 8.1 Enum 类集中管理 priority/type/status 等枚举值，避免散落在多处的字符串不一致问题再次发生

## 二、问题详情与修复

### #1 [P2] ChangeManager 8 个字段无验证规则

**文件：** `app/Livewire/Itsm/ChangeManager.php`
**问题：** `$rules` 仅定义了 `formTitle` 和 `formProjectId` 两个验证规则，但 `save()` 方法写入数据库的字段包括：

| 字段 | 写入 DB | 验证规则 | 问题 |
|------|---------|---------|------|
| formTitle | ✅ | required\|string\|max:200 | ✅ 已验证 |
| formProjectId | ✅ | required\|exists:projects,id | ✅ 已验证 |
| formServiceId | ✅ | ❌ 无 | 可注入不存在的 service_id |
| formType | ✅ | ❌ 无 | 可注入任意 type 值（如 'xxx'） |
| formRisk | ✅ | ❌ 无 | 可注入任意 risk 值 |
| formDescription | ✅ | ❌ 无 | 无长度限制，可写入超长文本 |
| formRollbackPlan | ✅ | ❌ 无 | 同上 |
| formWindowStart | ✅ | ❌ 无 | 无日期格式验证 |
| formWindowEnd | ✅ | ❌ 无 | 无日期格式验证，无开始/结束顺序校验 |

**影响：** 用户可通过篡改前端请求注入任意 type/risk 字符串（破坏数据一致性）、超长描述（DB 存储压力）、非法日期值（破坏变更窗口逻辑）。

**修复：** 补充完整验证规则：
```php
protected $rules = [
    'formTitle'        => 'required|string|max:200',
    'formProjectId'    => 'required|exists:projects,id',
    'formServiceId'    => 'nullable|exists:services,id',
    'formType'         => 'required|in:release,config,rollback,hotfix',
    'formRisk'         => 'required|in:low,medium,high,critical',
    'formDescription'  => 'nullable|string|max:5000',
    'formRollbackPlan' => 'nullable|string|max:5000',
    'formWindowStart'  => 'nullable|date',
    'formWindowEnd'    => 'nullable|date|after_or_equal:formWindowStart',
];
```

### #2 [P2] IncidentManager 4 个字段无验证规则

**文件：** `app/Livewire/Itsm/IncidentManager.php`
**问题：** 与 #1 同类问题。`$rules` 仅验证 `formTitle` 和 `formProjectId`，但 `save()` 写入的 `formSeverity`、`formDescription`、`formServiceId`、`formAssignedTo` 均无验证。

**影响：**
- `formSeverity` 可注入任意值（如 'P0'、'xxx'），与模型 `getSeverityLabelAttribute` 的 P1-P4 匹配不一致
- `formDescription` 无长度限制
- `formServiceId` 可注入不存在的 service_id（孤立外键引用）
- `formAssignedTo` 可注入不存在的 user_id

**修复：** 补充验证规则：
```php
protected $rules = [
    'formTitle'      => 'required|string|max:200',
    'formProjectId'  => 'required|exists:projects,id',
    'formServiceId'  => 'nullable|exists:services,id',
    'formSeverity'   => 'required|in:P1,P2,P3,P4',
    'formDescription'=> 'nullable|string|max:5000',
    'formAssignedTo' => 'nullable|exists:users,id',
];
```

### #3 [P2] IDOR 漏洞 — 项目详情/看板/任务管理无访问控制

**文件：** `app/Livewire/Projects/ProjectDetail.php`、`app/Livewire/Projects/TaskKanban.php`、`app/Livewire/Projects/TaskManager.php`

**问题：** 三个组件的 `mount(Project $project)` 方法均通过路由模型绑定接收 Project 实例，但**无任何访问控制检查**。路由定义也未加 `can` 中间件：

```php
Route::get('/{project}', ProjectDetail::class)->name('show');           // 无 can 中间件
Route::get('/{project}/kanban', TaskKanban::class)->name('kanban');    // 无 can 中间件
```

**影响：** 任意已认证用户可通过修改 URL 中的 project ID（如 `/projects/42`），查看**非成员项目**的：
- 项目详情（标题、描述、进度、成员列表、负责人）
- 所有任务（标题、分配人、优先级、状态、评论）
- 操作日志（谁在何时做了什么变更）
- 附件信息

这构成 **IDOR (Insecure Direct Object Reference)** 漏洞。系统的权限模型区分了 `view projects`（查看参与的项目）和 `view all projects`（查看所有项目），但此区分在项目详情页完全失效。

**修复：** 在三个组件的 `mount()` 中添加访问控制：
```php
$user = auth()->user();
if (!$user->can('view all projects')) {
    $isMember = $project->members()->where('user_id', $user->id)->exists()
        || (int) $project->created_by === $user->id;
    if (!$isMember) {
        abort(403, __('无权访问此项目。'));
    }
}
```

**注意：** `ProjectList` 和 `MyProjects` 组件已正确实现了项目范围过滤（非管理员只看成员项目），但项目详情页的 IDOR 绕过了列表层的访问控制。

### #4 [P3] .env 缺失 SESSION_SECURE_COOKIE

**文件：** `.env`
**问题：** `.env.example` 定义了 `SESSION_SECURE_COOKIE=false`（注释说明 HTTPS 环境必须设为 true），但 `.env` **遗漏了此配置项**。

Laravel `config/session.php` 中 `'secure' => env('SESSION_SECURE_COOKIE')`，当 env 未设置时返回 `null`（非 `false`），Laravel 会根据请求是否 HTTPS 自动判断。但显式设置更清晰，且与 `.env.example` 保持一致。

**修复：** 在 `.env` 中补充 `SESSION_SECURE_COOKIE=false`（本地开发 HTTP 保持 false，生产 HTTPS 改为 true）。

### #5 [P3] Dashboard recentLogs 三层嵌套查询优化

**文件：** `app/Livewire/Dashboard.php`
**问题：** 非管理员的最近日志查询使用三层嵌套 `whereHas`：
```php
ProjectLog::with(['user', 'project'])
    ->whereHas('project', function ($sub) use ($user) {
        $sub->whereHas('members', fn($m) => $m->where('user_id', $user->id))
            ->orWhere('created_by', $user->id);
    })
```
这会产生三层嵌套子查询 SQL（`WHERE EXISTS (SELECT ... WHERE EXISTS (SELECT ...))`），在项目/成员/日志数据量增大时性能显著下降。

**修复：** 改为先查出用户参与的 project_ids（1-2 次简单查询），再用 `whereIn` 过滤日志：
```php
$myProjectIds = $user->assignedProjects()->pluck('project_id')
    ->merge(Project::where('created_by', $user->id)->pluck('id'))
    ->unique()->values()->toArray();
$q->whereIn('project_id', $myProjectIds);
```
`whereIn` 使用 IN 子句（单层查询），比嵌套 EXISTS 性能更好，且 project_ids 数组通常较小（用户参与的项目有限）。

## 六、审查中确认良好的部分（无需修改）

- **Mass Assignment 防护**：所有 21 个 Model 均使用 `$fillable` 白名单，无 `$guarded = []` 或无保护的模型
- **Blade XSS 防护**：`{!! !!}` 仅用于 `__()` 翻译函数（开发者控制的静态字符串），用户数据均通过 `{{ }}` 自动转义
- **ChangeManager 职责分离**：approve() 检查不能批准自己提交的变更（`requester_id === auth()->id()` 拒绝）
- **ChangeManager 状态守卫**：edit() 仅允许编辑 draft/rejected 状态，submitForApproval/approve/reject/complete/rollback 均有状态检查
- **Session 加密**：SESSION_ENCRYPT=true 已启用
- **Session same_site**：默认 'lax' 防 CSRF
- **Session http_only**：默认 true 防 JS 读取
- **ProjectList 范围过滤**：非管理员只看成员项目（`whereHas('members')` + `orWhere('created_by')`）
- **KnowledgeBase 搜索转义**：`addcslashes($search, '%_')` 防 LIKE 通配符注入
- **TicketBoard 外键验证**：formProjectId/formAssetId/formAssignedTo 均有 `exists` 验证

## 二、问题详情与修复

### #1 [P2] ServiceManager 6 个字段无验证规则

**文件：** `app/Livewire/Itsm/ServiceManager.php`
**问题：** `$rules` 仅验证 `formName` 和 `formType`，但 `save()` 写入数据库的字段包括 `formProjectId`、`formStatus`、`formDescription`、`formOwnerId`、`formHealthUrl`、`formTags`，均无验证。

这是第6轮 #1（ChangeManager）和 #2（IncidentManager）同类问题的延续 — ServiceManager 也存在验证规则覆盖不全的问题。

| 字段 | 写入 DB | 验证规则 | 问题 |
|------|---------|---------|------|
| formName | ✅ | required\|string\|max:100 | ✅ 已验证 |
| formType | ✅ | required\|string | ✅ 已验证（但缺 max 限制） |
| formProjectId | ✅ | ❌ 无 | 可注入不存在的 project_id |
| formStatus | ✅ | ❌ 无 | 可注入任意 status 值 |
| formDescription | ✅ | ❌ 无 | 无长度限制 |
| formOwnerId | ✅ | ❌ 无 | 可注入不存在的 user_id |
| formHealthUrl | ✅ | ❌ 无 | 无 URL 格式验证 |
| formTags | ✅ | ❌ 无 | 无长度限制 |

**修复：** 补充完整验证规则：
```php
protected $rules = [
    'formName'        => 'required|string|max:100',
    'formType'        => 'required|string|max:50',
    'formProjectId'   => 'nullable|exists:projects,id',
    'formStatus'      => 'required|in:healthy,degraded,down,maintenance,unknown',
    'formDescription' => 'nullable|string|max:2000',
    'formOwnerId'     => 'nullable|exists:users,id',
    'formHealthUrl'   => 'nullable|url|max:500',
    'formTags'        => 'nullable|string|max:500',
];
```

### #2 [P2] 企微/钉钉通讯录同步无频率限制

**文件：** `app/Livewire/Admin/ImSettings.php`
**问题：** `syncWechatUsers()` 和 `syncDingtalkUsers()` 方法无任何频率限制。管理员可无限次点击同步按钮，导致：
- 企微/钉钉 API 调用频率超限 → access_token 被限流
- 频繁全量拉取通讯录 → 应用可能被平台封禁
- 每次同步创建/更新大量用户记录 → DB 负载

**修复：** 基于 Cache 的限流，每管理员每小时最多 3 次同步操作（企微和钉钉共享计数）：
```php
$rateKey = 'im_sync_rate:' . auth()->id();
$syncCount = (int) Cache::get($rateKey, 0);
if ($syncCount >= 3) {
    $this->syncResult = __('同步操作过于频繁，每小时最多 3 次，请稍后再试。');
    return;
}
// ... 同步逻辑 ...
Cache::put($rateKey, $syncCount + 1, 3600);
```

### #3 [P2] AD 手动同步无频率限制

**文件：** `app/Livewire/Admin/AdSettingsManager.php`
**问题：** `syncNow()` 方法无频率限制。管理员可频繁点击同步按钮，每次执行 `ad:sync-users` 命令会：
- 建立 LDAP 连接并遍历所有 AD 用户
- 对每个用户执行 DB 查询和更新/创建操作
- 频繁执行可能导致 AD 服务器负载过高

自动调度（routes/console.php 中的 Schedule）不受此限，仅限制手动触发。

**修复：** 基于 Cache 的限流，每管理员每小时最多 5 次手动同步：
```php
$rateKey = 'ad_sync_rate:' . auth()->id();
$syncCount = (int) Cache::get($rateKey, 0);
if ($syncCount >= 5) {
    $this->syncStatus  = 'fail';
    $this->syncMessage = __('手动同步操作过于频繁，每小时最多 5 次，请稍后再试。');
    return;
}
```

### #4 [P3] KnowledgeBase 权限拒绝静默返回

**文件：** `app/Livewire/Itsm/KnowledgeBase.php`
**问题：** `delete()` 和 `deleteAttachment()` 方法在权限不足时静默 `return`，不设置任何 `session()->flash()` 消息。这与第3轮 #6（AssetManager）和第6轮发现的问题一致 — 用户点击删除后无法区分"删除成功"和"权限不足被拒绝"。

**修复：** 统一为 `session()->flash('kb_error', ...)` + `return`，并在成功时补充成功提示。

---

### #5 [P3] IncidentManager addTimeline timelineNote 无长度限制

**文件：** `app/Livewire/Itsm/IncidentManager.php`
**问题：** `addTimeline()` 方法直接将 `$this->timelineNote` 写入 DB 的 `description` 字段，无任何长度验证。用户可注入超长文本（数 MB），导致 DB 存储压力和查询性能下降。

**修复：** 补充 5000 字长度限制，与 KnowledgeBase 评论（5000 字）、TaskManager 评论（5000 字）一致：
```php
if (mb_strlen($this->timelineNote) > 5000) {
    session()->flash('error', __('时间线备注不能超过 5000 字'));
    return;
}
```

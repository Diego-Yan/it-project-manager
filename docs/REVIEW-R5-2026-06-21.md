### #1 [P3] 引入 PHP 8.1 Enum 集中管理枚举

**文件：** 新建 `app/Enums/TicketPriority.php`、`app/Enums/TicketStatus.php`；修改 `app/Models/Ticket.php`

**背景：** 第2-4轮审查多次发现枚举不一致问题：
- 第2轮 #1：Ticket priority 验证用 `urgent`，但 Blade/模型/SLA 用 `critical` → 紧急工单无法创建
- 第2轮 #5：Project type 常量 `maintenance`/`upgrade` 与实际 `new`/`improved`/`issue` 不符
- 第4轮 #1：Zabbix 灾难级告警映射 high 而非 critical

这些问题的根因是枚举值散落在 10+ 个文件的裸字符串中，缺乏单一事实来源。

**修复：** 引入 PHP 8.1 backed enum 作为枚举的权威定义：

**TicketPriority enum** 提供：
- 4 个枚举值：Low/Medium/High/Critical
- `label()` / `color()` / `sortOrder()` 方法
- `validationRule()` 生成 `'in:low,medium,high,critical'`
- `values()` 返回值数组

**TicketStatus enum** 提供：
- 4 个枚举值：Open/InProgress/Resolved/Closed
- `label()` / `color()` 方法
- `allowedTransitions()` / `canTransitionTo()` 状态机方法（替代 Ticket 模型中原有的常量数组）
- `validationRule()` / `values()` 方法

**Ticket 模型** 添加 enum cast：
```php
'priority' => \App\Enums\TicketPriority::class,
'status'   => \App\Enums\TicketStatus::class,
```

**注意：** 本轮为渐进式引入，仅创建 Enum 类和 Ticket 模型 cast。现有代码中的裸字符串（如 `'critical'`）因 enum cast 会在模型访问时自动转换为 enum 实例，Blade 模板中 `$ticket->priorityLabel` 等访问器仍正常工作（因为 priority 字段 cast 后，`getPriorityLabelAttribute` 中的 `match($this->priority)` 会匹配 enum 实例的 value）。未来各组件可逐步迁移为引用 enum 常量。

### #2 [P2] SSRF 防护 — 服务器出站请求 URL 无内网校验

**文件：** 新建 `app/Services/SsrfGuard.php`；修改 `WebhookManager.php`、`AiSettings.php`、`ZabbixManager.php`

**问题：** 系统有三处服务器端 HTTP 出站请求，URL 由管理员配置，无任何内网地址校验：

| 场景 | 文件 | 风险 |
|------|------|------|
| Webhook 投递 | SendWebhookNotification.php → 读取 WebhookConfig.url | 管理员配置内网地址 → 服务器向内网发送请求 |
| LLM API 调用 | AiChat.php → config('services.llm.url') | 管理员配置 169.254.169.254 → 探测云元数据 |
| Embedding 测试 | AiSettings.php → $this->embeddingUrl | 同上 |
| Zabbix API 调用 | ZabbixService.php → $this->config->url | Zabbix 通常在内网，需区别对待 |

**SSRF 攻击面：**
- 云环境：`169.254.169.254` 元数据服务 → 窃取 IAM 凭证
- 内网探测：`192.168.x.x` / `10.x.x.x` → 端口扫描/服务探测
- 本地服务：`127.0.0.1` → 访问仅本机可达的服务

**修复：** 新建 `SsrfGuard` 工具类，提供 `isSafe($url)` 方法：
- 解析 URL 的 host → DNS 解析为 IP
- 校验 IP 是否属于内网/保留地址段（10.x / 172.16-31.x / 192.168.x / 169.254.x / 127.x / 0.x / IPv6 loopback）
- 使用 PHP `filter_var` + `FILTER_FLAG_NO_PRIV_RANGE | FILTER_FLAG_NO_RES_RANGE`

**应用策略（分层）：**
- **Webhook URL**：全量阻止内网地址（webhook 应投递到外部企微/钉钉服务）
- **LLM/Embedding API URL**：全量阻止内网地址（LLM 是外部云服务）
- **Zabbix URL**：仅阻止 link-local（169.254.x）和 0.0.0.0，**允许内网地址**（Zabbix 通常部署在内网，与外部 SaaS 不同）

### #3 [P2] 日志驱动 single 无轮转 → laravel.log 无限增长

**文件：** `.env`、`.env.example`

**问题：** 日志配置为 `LOG_STACK=single`，Laravel `single` 驱动将所有日志写入单一 `storage/logs/laravel.log` 文件，**无轮转、无大小限制**。长期运行后该文件会无限增长，可能写满磁盘导致服务中断。

此外，`routes/console.php` 中多个 `Schedule::command(...)->appendOutputTo(storage_path('logs/xxx.log'))` 也是无轮转追加写入，进一步加剧磁盘增长。

**修复：**
- `.env` 和 `.env.example` 中 `LOG_STACK=single` → `LOG_STACK=daily`
- `daily` 驱动按日期分割为 `laravel-2026-06-21.log` 等文件，默认保留 14 天（`LOG_DAILY_DAYS=14`），自动清理旧日志

**注意：** `appendOutputTo` 写入的调度日志（ad-sync.log / deadlines-check.log 等）仍为无轮转追加。这些文件通常较小（调度命令输出有限），且为 Laravel scheduler 的原生行为，全面改造需自定义日志通道，本轮标记为后续优化项。

### #4 [P2] CSP connect-src 文档说明

**文件：** `app/Http/Middleware/SecurityHeaders.php`

**问题：** 原 CSP 配置 `connect-src 'self'` 实际已正确允许 Livewire 的同源请求（Livewire 向 `/livewire/update` 发送 POST）。但缺乏注释说明为何 `connect-src` 设为 `'self'` 以及外部 LLM API 调用为何不受 CSP 影响，可能误导后续开发者误改为允许外部地址。

**修复：** 补充详细注释说明：
- `connect-src 'self'` 允许 Livewire 同源请求
- AiChat 调用外部 LLM API 在服务端（PHP `Http::post`），不经过浏览器 CSP
- 不需要在 `connect-src` 中放行外部 LLM 地址

## 七、审查中确认良好的部分（无需修改）

- **ZabbixConfig Token 加密**：Crypt::encryptString 存储，防重复加密，掩码显示，编辑不暴露
- **AiSettings 敏感字段保护**：dehydrate() 清除 key，不预填充，guard() 权限检查
- **KbAttachment 文件名安全**：preg_replace 过滤特殊字符，仅保留字母数字中文点号下划线横线
- **KbAttachment MIME 自动检测**：preview_type 从 mime_type 智能推断，null guard 完善
- **notifications 表索引**：(user_id, is_read) + (user_id, created_at) 双索引覆盖
- **NotificationBell 查询优化**：2 次查询合并为 1 次
- **AdSyncUsers 无大小限制**：sizelimit=0 避免 AD 用户丢弃
- **Webhook Job 异步化**：通过 dispatch() 投递队列，不阻塞 UI
- **Webhook 单点失败隔离**：try/catch 单个 webhook 失败不阻塞其余投递
- **CSP 安全头**：CSP/X-Frame-Options/X-Content-Type-Options/Referrer-Policy/Permissions-Policy/HSTS 配置完善

## 八、后续建议（第6轮可关注）

1. **Enum 全面迁移**：将各组件中的裸字符串枚举（如 `'critical'`、`'in_progress'`）逐步替换为 `TicketPriority::Critical->value` 等引用，彻底消除字符串不一致风险
2. **知识库附件访问控制**：附件存储在 public disk，建议改为受控下载路由（需认证 + 文章可见性检查）
3. **Dashboard recentLogs 查询优化**：非管理员的嵌套 whereHas 可能产生低效 SQL，建议用 whereIn + 预查 project_ids
4. **调度日志轮转**：appendOutputTo 写入的日志无轮转，建议为每个调度命令配置独立 daily 日志通道
5. **Webhook 投递日志记录**：当前仅记录成功/失败日志，建议记录到数据库表供管理员查看投递历史
6. **密码策略配置化**：UserManager 的密码复杂度规则硬编码在验证规则中，建议提取为 config 可配置

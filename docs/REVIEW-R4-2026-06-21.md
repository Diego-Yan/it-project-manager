## 二、问题详情与修复

### #1 [P2] Zabbix 灾难级告警优先级映射缺失 critical

**文件：** `app/Console/Commands/ZabbixPoll.php` 第 74 行
**问题：** Zabbix 告警自动转工单时，优先级映射为：
```php
'priority' => $severity >= 4 ? 'high' : ($severity >= 3 ? 'medium' : 'low'),
```
Zabbix severity 5 = 灾难（Disaster），但映射为 `high` 而非 `critical`。这导致：
- 灾难级告警永远无法触发 critical 级别 SLA
- 第3轮补充的 SLA Seeder 中 critical 级别 SLA（响应15min/解决2h）对 Zabbix 告警无效
- 仪表盘和工单列表无法按最高优先级筛选灾难级告警

**修复：** 补充 `severity >= 5 → critical` 分支：
```php
'priority' => $severity >= 5 ? 'critical' : ($severity >= 4 ? 'high' : ($severity >= 3 ? 'medium' : 'low')),
```

### #2 [P2] Region Seeder 缺失 — 新部署工单表单无法提交

**文件：** `database/seeders/DatabaseSeeder.php`
**问题：** `TicketBoard` 和 `MyTickets` 的 `formRegionId` 验证规则为 `required|exists:regions,id`。但 `DatabaseSeeder` **未创建任何 Region 数据**。

**影响：** 全新部署 `php artisan migrate --seed` 后，用户在工单表单中选择地区时下拉框为空，由于 `required|exists` 验证，**工单表单无法提交**。工单创建功能在新部署环境中完全不可用。

**修复：** 在 Seeder 中补充杭州/深圳双地区数据（与 yanmade.com 实际两地办公一致）：
```php
$regions = [
    ['name' => '杭州', 'sort_order' => 1],
    ['name' => '深圳', 'sort_order' => 2],
];
```

### #3 [P1] SLA Seeder 缺失 — SLA 违约检测完全失效

**文件：** `database/seeders/DatabaseSeeder.php`
**问题：** `DatabaseSeeder` 创建了 SLA 相关的权限（`view slas`、`manage slas`），但**未创建任何 SLA 数据记录**。`slas` 表迁移定义了 `priority` 字段（`low|medium|high|critical`），但无初始数据。

**影响链路：**
1. `Sla::getDeadline($priority)` 查询 `where('priority', $priority)->where('is_active', true)->first()`
2. 无匹配记录 → 返回 `null`
3. 工单创建时 `sla_deadline = null`
4. `Ticket::isSlaBreached()` → `$this->sla_deadline && ...` → `null && ...` → `false`
5. **所有工单永远不会被标记为 SLA 违约**，SLA 超时监控完全失效
6. 即使第2轮修复了 priority 枚举为 `critical`，无 SLA 数据则 `critical` 级别的 15min 响应/2h 解决时限也不生效

**修复：** 补充 4 个优先级的默认 SLA 配置：

| 优先级 | 响应时限 | 解决时限 | 说明 |
|--------|---------|---------|------|
| low | 8h (480min) | 2天 (2880min) | 低优先级问题 |
| medium | 4h (240min) | 1天 (1440min) | 标准请求 |
| high | 1h (60min) | 8h (480min) | 紧急问题 |
| critical | 15min | 2h (120min) | 灾难级故障 |

### #4 [P3] UserManager 用户列表未排除 password 字段

**文件：** `app/Livewire/Admin/UserManager.php` 第 368 行
**问题：** `render()` 中的用户列表查询 `User::with('roles')` 未显式 `select()` 字段，默认查询所有列（含 `password`、`remember_token`）。

User 模型 `$hidden = ['password', 'remember_token']` 会阻止 `toArray()/JSON` 序列化，但 Livewire 将 paginator 对象传入 Blade 后，若模板中使用 `{!! !!}` 或 JSON 输出仍可能泄露。显式 select 最小字段集是纵深防御。

**修复：** 添加 `->select([...])` 仅查询必要字段，排除 `password` 和 `remember_token`。

### #5 [P3] failed_jobs 表无定时清理

**文件：** `routes/console.php`
**问题：** 队列 Job 重试 3 次失败后写入 `failed_jobs` 表（第3轮 #4 修复后此路径会被触发）。但无清理机制，该表会无限增长。

**修复：** 添加每日 4:00 清理 7 天前失败 Job 的调度：
```php
Schedule::command('queue:prune-failed --hours=168')
    ->dailyAt('04:00')
    ->withoutOverlapping()
    ->appendOutputTo(storage_path('logs/queue-prune.log'));
```
保留 7 天近期记录供排障，清理历史数据。

### #6 [P2] AiChat LLM 调用无频率限制

**文件：** `app/Livewire/AiChat.php`
**问题：** `send()` 方法无任何频率限制。用户（或脚本）可无限次调用外部 LLM API，导致：
- 外部 API 成本失控（按 token 计费的模型尤其严重）
- API 配额耗尽影响其他功能
- 潜在的 DoS 放大攻击面

**修复：** 基于 Cache 的双层限流：
- **每分钟最多 10 次**（防止突发刷量）
- **每小时最多 60 次**（防止持续滥用）
超限时返回友好提示，不调用 LLM。

### #7 [P2] 消耗品 asset_tag 字典序编号 Bug

**文件：** `app/Livewire/Itsm/AssetManager.php`
**问题：** 消耗品自动编号使用 `max('asset_tag')` 对字符串做字典序比较：
```php
$maxTag = $query->max('asset_tag');  // 返回字符串
$nextNum = $maxTag ? (int)substr($maxTag, 4) + 1 : 1;
```
当消耗品超过 9999 条时，`CON-10000` 的字典序**小于** `CON-9999`（因为 '1' < '9'），`max()` 返回 `CON-9999` 而非 `CON-10000`，导致下一个编号回退到 `CON-10000`，与已有记录冲突。

**修复：** 改为提取数字部分做整数比较：
```php
$allTags = $query->pluck('asset_tag')->filter()->values();
$maxNum = 0;
foreach ($allTags as $tag) {
    $num = (int) substr($tag, 4);
    if ($num > $maxNum) $maxNum = $num;
}
$nextNum = $maxNum + 1;
```
注意：对于超大型数据集，可在 DB 层用 `CAST(SUBSTR(asset_tag, 5) AS INTEGER)` 优化，但当前 `pluck + PHP 比较` 方案对 SQLite/MySQL 均兼容，且消耗品数量通常不会达到性能瓶颈。

## 六、审查中确认良好的部分（无需修改）

- **ZabbixConfig Token 加密**：`Crypt::encryptString` 存储，`getMaskedTokenAttribute` 前端掩码，编辑时不暴露真实 token
- **AiSettings 敏感字段保护**：`dehydrate()` 清除 key，不在 mount 中预填充，`guard()` 权限检查
- **NotificationBell 查询优化**：2 次查询(get+count) → 1 次查询，未读数从 collection 计算
- **ZabbixPoll 去重机制**：24h 内同 triggerId 不重复创建工单
- **ZabbixPoll 系统用户兜底**：配置用户不存在时 fallback 到首个活跃用户
- **AdSyncUsers 无大小限制**：`sizelimit=0` 避免 AD 用户被静默丢弃
- **notifications 表索引**：`(user_id, is_read)` + `(user_id, created_at)` 双索引覆盖查询场景
- **DailyDigest 统计查询**：单次 count 查询，无 N+1

## 七、后续建议（第5轮可关注）

1. **PHP Enum 类引入**：第2-4轮多次发现枚举不一致问题（priority urgent/critical、Project type 常量、Zabbix severity 映射），建议用 PHP 8.1 backed enum 集中管理
2. **知识库附件访问控制**：附件存储在 public disk，知道 URL 即可无认证访问，建议改为受控下载路由
3. **Dashboard recentLogs 查询**：非管理员的 `whereHas('project', fn($sub) => ...)` 嵌套可能产生低效 SQL，建议用 `whereIn` + 预查 project_ids
4. **SSRF 防护**：AiChat.callLlm() 和 ZabbixService 的请求 URL 来自管理员配置，建议增加内网地址校验
5. **Webhook URL SSRF 防护**：WebhookManager 保存的 URL 无内网地址校验，管理员可配置内网地址探测
6. **日志轮转配置**：`storage/logs/*.log` 由 scheduler `appendOutputTo` 写入，无轮转配置可能写满磁盘
